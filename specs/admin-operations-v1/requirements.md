# 唐小识运营中心 V1 需求

## 目标

将现有审核工具升级为适合小团队使用的无人车运营后台。Web 承担完整运营与小程序内容管理，管理员小程序承担移动待办；本地和线上统一使用 Supabase、TOS 与微信通知能力。

## 用户与边界

- 超级管理员管理全部业务与配置；运营处理订单和咨询；财务只查看支付、退款与金额。
- 按趟订单在线提交、审核和支付，管理员在线下安排无人车，不管理司机或单车定位。
- 包月与租购作为销售咨询，不进入按趟订单状态机。
- 车型数量是软提示，不阻断审核；本地图片上传继续禁用，完成凭证使用演示记录。

## 验收需求

1. 当管理员进入 Web 后台时，系统应按“概览、订单履约、客户跟进、小程序运营、系统管理”展示分组侧边栏，每个业务页面独立呈现。
2. 当管理员查看工作台时，系统应展示待审核订单、待联系包月/租购咨询、车型可用数量和数量不足提醒。
3. 当超级管理员保存车型时，系统应保存总数量，并返回预占数量和参考可用数量。
4. 当按趟订单审核通过时，系统应按管理员填写的车辆数建立预占；数量不足时应提醒但不阻断操作。
5. 当报价失效、付款前取消、退款取消或订单完成时，系统应释放该订单的车辆预占。
6. 当订单已付款时，有权限的管理员应能依次推进为调度中、配送中和已完成；安排车辆时填写派车数量并可选填车牌，完成时暂不要求图片凭证。
7. 当新订单、新咨询、预约临近、车辆不足或退款待处理发生时，系统应生成管理员通知；Web 和管理员小程序均可查看并标记已读。
8. 当管理员处理包月或租购咨询时，系统应支持按类型和状态筛选并更新跟进状态及备注。
9. 当超级管理员维护小程序内容时，车型、轮播、计费、文案联系方式和管理员授权应分别位于独立页面。
10. 所有新增行为应使用统一的 Supabase 数据源和 API 信封，并保留审计日志。
11. 当管理员查看首页时，系统应只保留关键指标与一行横向滚动的按趟车型数量，不展示最近待审核列表和咨询待办卡片。
12. 当管理员切换订单状态筛选时，已选择的日期范围应保持不变。
13. 当管理员查看订单详情时，系统应先展示下单、审核、配车、配送、完成流程图，再按左地址右订单信息、全宽物品信息组织内容，并提供地址复制操作。
14. 当订单待审核、待配车或待完成时，对应主按钮应始终可见，不依赖鼠标悬停；拒绝时应提供常用原因快捷选项。
15. 当订单地址包含有效坐标时，系统应按两点坐标计算参考路线距离并自动填入审核报价，管理员仍可修改。
16. 当订单已拒绝时，详情底部应独立展示拒绝原因；订单审核人应显示管理员昵称，昵称缺失时回退用户名。
17. 当管理员查看包月或租购咨询时，页面应删除重复的类型选择，仅保留状态筛选；详情标题展示公司与联系人，电话可复制。
18. 当管理员查看车型与数量时，按趟、包月、租购三个模式选择应保持在同一行。
